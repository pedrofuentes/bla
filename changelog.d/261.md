- Snippets settings tab + CRUD IPC commands (M4, issue #261, AC-54): new
  thin `#[tauri::command]` wrappers `list_snippets`/`add_snippet`/
  `update_snippet`/`remove_snippet` over the `Store` CRUD from #258, and a
  real `SnippetsTab.tsx` replacing the settings window's "coming soon"
  Snippets placeholder. Each entry renders as an inline-editable
  trigger/body pair (a text input + a `<textarea>`, since a snippet's body
  can be multi-line); an add form (client-side rejects a blank trigger/body
  or a case-insensitive duplicate trigger before ever calling `add_snippet`,
  since — like `add_dictionary_term` — the backend's `UNIQUE COLLATE NOCASE`
  constraint makes a duplicate a silent no-op, not a rejected call).
  Per-row trigger/body edits commit on blur rather than per-keystroke
  (mirroring `GeneralTab.tsx`'s base-folder/path-template fields), guarded
  by a per-row generation counter against out-of-order responses (the same
  pattern `ToneTab.tsx`'s tone-edit uses) so a slow stale response can never
  clobber a faster newer one to the same row; unlike `add_snippet`,
  `update_snippet` genuinely rejects a trigger colliding with a different
  row (the schema constraint applies on UPDATE too), surfaced as a
  row-scoped, kind-only inline error. Every rendered error is a static,
  kind-derived string — never a raw backend message — and neither triggers
  nor bodies are ever `console.log`ged. No store schema change (done in
  #258) and no snippet-matching/pipeline wiring (done in #260, wired to the
  pipeline separately).
