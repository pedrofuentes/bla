- Fixed the recording pill/tray icon being able to get stuck out of sync with the actual pipeline state (issue #128, escalated by Sentinel on PR #127): `apply_pipeline_state` wrote the shared pipeline state, computed the pill's visibility, and enqueued a main-thread closure to apply both — three steps that weren't atomic as a unit, so two same-generation state transitions (e.g. one from the hotkey thread, one from the pipeline thread) could have their main-thread closures run in the opposite order from the state writes that produced them. The older closure, applying a value it captured at enqueue time, could then run *after* the newer one — leaving the pill stuck visible while the pipeline was already Idle, or hidden while it was actually Recording, until an unrelated later transition happened to correct it. `AppState` now holds pipeline state and pill visibility together as one `tray::PipelineDisplay` behind a single mutex, and the main-thread closure no longer carries a captured snapshot at all — it re-derives what to show (`tray::resolve_display`) from whatever `AppState` currently holds at the moment it actually runs. However many closures end up executing, in whatever order, each one applies the same current truth, so the race is closed regardless of ordering rather than papered over for the common case.
