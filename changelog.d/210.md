- Two Sentinel follow-ups fixed together (issues #210, #220): the Settings → Output "Base folder (vault)" field now validates absoluteness client-side (`src/lib/baseDir.ts`'s `validateBaseDir`, mirroring `src/lib/pathTemplate.ts`'s inline-error/withhold-on-invalid UX) — `output::resolve_base_dir` uses the configured string verbatim (never expanding `~`, never resolving a relative path against anything but the process's CWD at write time), so a relative value previously wrote into an unexpected, launch-inconsistent location with no warning; the validator accepts either platform's absolute form (leading `/`, or a Windows drive-letter `C:\`/`C:/` or UNC `\\server\share` root) since settings.json could sync across platforms, and rejects everything else including a bare `~` the backend never expands (#210). Separately, `Store::insert_history` failures at the `run_pipeline_in_background` call site — previously `eprintln!`-only, invisible in a packaged GUI and a silent history-row loss — now also emit a new, informational `errors::ErrorKind::HistoryPersistFailed` through the existing typed `pipeline-error` event surface (kind-only, no data from the underlying `rusqlite::Error`), which the pill renders as a non-blocking "Couldn't save this dictation to history." toast (`toastForError`, alongside the existing `OllamaUnreachable` case); the row insert itself stays unconditional (never skipped for a stale generation, matching #214/#218's existing never-drop-the-row rule) but the toast itself is gated behind `generation_is_live` like every other UI-visible effect in that function, via a new pure `tray::should_settle_with_notice` decision that now covers both the Ollama-fallback and history-persist-failure notice cases (#220).
