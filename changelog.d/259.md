- Command-mode hotkey wiring (M4 backbone, issue #259, part of #242, AC-23/AC-49):
  a second, independently-configured global hotkey
  (`settings::Settings::command_hotkey`, default `Control+Shift+C`) drives its
  own `hotkeys::StateMachine` instance — a separate hold/toggle press/release
  session from the dictation hotkey's, sharing the same persisted
  `recording_mode`. Pressing it: captures the active selection
  (`output::capture_selection`, #257), records the spoken instruction, reuses
  `stt.rs`'s existing `Stt::transcribe` to transcribe it, runs #256's
  `OllamaCommand::transform` over the two as separate CONTENT/INSTRUCTION-
  channel arguments (never concatenated), and pastes the result back over the
  selection (`output::replace_selection`, #257) — restoring the user's
  original clipboard in every failure branch (unreachable transport, empty
  selection, blank instruction, paste failure), never just a kind-only UI
  notice with the clipboard left holding stale mid-flow content. The new pure
  orchestration (`run_command_transform`, generic over `Stt`/
  `CommandTransform`/`Clipboard`/`PasteSynthesizer`, no `AppState`) is
  unit-tested with fakes for the full happy path and every safe-degradation
  branch (AC-23/AC-47). `hotkeys::distinct_hotkeys` (AC-49) rejects a settings
  save that would leave the dictation and command-mode hotkeys bound to the
  same accelerator (compared as parsed `Shortcut`s, so a differently-ordered
  spelling of the same chord is still caught), checked at `set_settings`-time
  before either hotkey is (re)registered or the settings blob is persisted.
  Command mode reuses (never forks) dictation's existing
  `AppState::dictation_generation`/pill-settle machinery, so an overlapping
  dictation and command-mode run can never clobber each other's pill state —
  no new tray/pill states.
  **Fix alongside this wiring:** the dictation hotkey's own (re-)registration
  (`register_hotkey`, and `commands::suspend_hotkey`'s capture-mode
  unregister) previously called `tauri-plugin-global-shortcut`'s
  `unregister_all()`, which was harmless when there was only ever one
  registered shortcut but would have silently dropped the new command-mode
  hotkey's live OS registration on the next dictation-hotkey rebind or
  settings-window hotkey-capture session; both now target-unregister only
  their own specific accelerator.
  No settings-window capture UI for the new field yet (#262); no
  history-row persistence of command-mode text (explicitly out of scope,
  not requested by #242).
