- Fixed a Sentinel finding on PR #245's cross-platform base-folder fix (issue #246): `validateBaseDir` (`src/lib/baseDir.ts`) previously accepted either platform's absolute syntax regardless of the runtime OS — a synced `settings.json` carrying a Windows `C:\...` form onto macOS (or a bare POSIX `/foo` onto Windows, where `Path::is_absolute` treats a leading `/` alone as drive-relative, not absolute) passed client-side validation even though `output::resolve_base_dir` runs Rust-side against THIS machine, reproducing #210's CWD-relative-write failure mode in that cross-platform edge case. `validateBaseDir` now takes the runtime platform as an explicit parameter and accepts only that platform's absolute form, rejecting a foreign-platform form with a distinct "Not an absolute path on this system…" inline error rather than folding it into the generic "must be absolute" message. The settings window fetches the runtime platform once on mount via a new, trivial, zero-argument `get_platform` command (`commands::get_platform`, a thin wrapper over a pure `cfg!(windows)` compile-time branch — Tauri never cross-compiles at runtime, so no OS syscall is involved) and passes it to the validator on every base-folder change/blur; a failed fetch (only realistically reachable in a test/harness environment that hasn't mocked the command) falls back to `"unix"`, this app's primary target. Same-platform accepts from #245 (and #210's original relative/tilde rejections) are unchanged and still covered — the platform matrix is exercised entirely client-side (`validateBaseDir(value, "windows" | "unix")` in `src/lib/baseDir.test.ts`), so no `#[cfg(windows)]`-gated Rust tests were needed.
