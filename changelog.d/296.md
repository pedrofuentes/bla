- Fixed command mode reporting "no usable selection" on Windows even when
  text was selected (ac7-p0, issues #296/#278). `output::capture_selection`
  synthesized the copy keystroke and then read the clipboard immediately,
  with no wait — but the focused app services the synthesized copy
  asynchronously, so on Windows the read raced ahead of it and returned the
  stale pre-copy clipboard value, which `captured_selection_is_usable` then
  rejected as empty. Capture now polls after the keystroke: it re-reads the
  clipboard up to a bounded number of times (default 8, ~40 ms apart, ~320 ms
  max), returning as soon as the value differs from the pre-copy contents —
  i.e. once the app has actually written the selection. The timing lives
  behind an injected sleep + poll-interval/count seam (mirroring
  `replace_selection`), so the retry loop is exercised deterministically in
  tests with a scripted clipboard and a zero interval, never a real clock.
  On genuine exhaustion (nothing was selected) the last read is returned
  as-is so the clear error still surfaces, with the clipboard untouched.
